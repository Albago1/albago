## What's already strong (don't touch)

- **Build health:** TypeScript compiles with zero errors; production build passes cleanly across all 48 routes.
- **Security:** Every admin route is guarded server-side with a role check (`app/admin/layout.tsx`); webhook endpoints require secrets (`x-webhook-secret`); the geocode proxy is properly designed with an LRU cache. No service-role key leaks into client code.
- **SEO plumbing:** Event pages have JSON-LD schema + OG metadata; the sitemap dynamically includes all published events/places/organizers; detail pages are server-rendered.
- **Craft details:** wizard draft persistence in localStorage, optimistic save-button updates, open-redirect protection on `/sign-in?next=`, submission rate limits, moderation queue with badge counts.

## 🔴 P0 — Fix immediately (trust & conversion killers)

### 1. `ID_Resend.txt` in repo root
Untracked file, likely a live Resend API key, flagged two sessions ago. One careless `git add .` publishes it.
**Fix:** Rotate the key in Resend, move the value to `.env.local`, delete the file.

### 2. Submit-event rejects users at step 8 of 8
Auth is only checked at final Submit (`lib/wizardSubmit.ts` → `submitCommunityEvent`). A signed-out visitor fills the entire wizard, then gets a bare error: *"Sign in to submit an event."* No link, no redirect, no mention that the draft is saved (it is, in localStorage — they just don't know). Single worst drop-off point in the product.
**Fix:** Check auth when the wizard mounts; show a friendly "sign in to continue — your draft will be waiting" gate with `?next=/submit-event`.

### 3. Developer error messages shown to end users
The submit flow surfaces strings like *"Database is missing Phase 25 RPC. Apply docs/seeds/phase-25-event-submission-rate-limits.sql in Supabase"* and *"GRANT EXECUTE may be missing"* directly in the UI (`lib/wizardSubmit.ts`).
**Fix:** Log details to console; show a generic "Something went wrong — try again or contact us."

### 4. `/submit-event` never verified end-to-end
Three column-cast hotfixes went in June 24 (`171ec7c`, `f9fd9b5`); the full `event_submissions` schema was never audited against the wizard payload. A fourth mismatch could still silently block real submissions.
**Fix:** Live test submission before anything else ships.

### 5. Uncommitted work aging in the tree
Edi Rama page copy rewrite (`lib/i18n/translations.ts`) + `.gitignore` `.vercel` addition, uncommitted since before June 30.
**Fix:** Commit or discard.

## 🟠 P1 — The two features that create love

### 6. Platform is English-first for an Albanian audience
A complete Albanian translation block exists (215 keys — more than English!), but only **10 of 73 client components** use the translation system. Hardcoded English everywhere else: the event wizard, events page filters ("Tonight", "This Weekend", "Featured first"), search dropdown ("Browse by city", "Search events, clubs, food..."), become-organizer, dashboard, every empty state. Switching to AL shows a half-translated site → reads as unfinished.
**Fix:** Systematic sweep wiring every user-facing string through `t()`.
Also: Spanish is missing 43 keys (all `er_*` Edi-Rama keys); language code `al` should ideally be `sq` (ISO) — cosmetic, fix during the sweep.

### 7. Organizer analytics don't exist at all
Zero view-tracking infrastructure — no `event_views` table, no tracking calls. The organizer dashboard shows only counts derived from the event list itself. **Views, saves, and share clicks per event over time** is what turns "we list your event" into "we help your event succeed" and brings organizers back weekly.
**Fix:** Lightweight tracking table + insert on event page view + sparklines in the dashboard (the `Sparkline` component already exists).

## 🟡 P2 — Flow friction

### 8. Wizard is 8 steps; two are single-choice screens
"Type" (event vs. protest) and "Category" are each one tap on their own step. Merging Type+Category into one screen and folding Media (optional anyway) into Basics → ~5 steps. Every removed step reduces abandonment.

### 9. Sign-up loses the return path
`/sign-in` honors `?next=` correctly; `/sign-up` hardcodes `/dashboard`. A user sent to sign-in mid-flow who clicks "create an account" loses their place.
**Fix:** One line per redirect in `app/sign-up/page.tsx`.

### 10. Main discovery page is fully client-rendered
`/events` ships an empty shell and fetches from the browser — "Loading events..." flash, slower first paint on mobile (primary platform), no server-rendered content for crawlers. Event *detail* pages are SSR'd properly; the list deserves the same.

### 11. Brand split confusion (product decision)
Homepage announces "Events · Nightlife · Civic Movements"; mobile nav gives Protests one of four tabs. Nightlife users land on protest banners; activists land among club listings. Decide deliberately: one identity, or clearer separation between the two worlds.

### 12. Small polish
- `window.confirm()` for wizard draft reset — use a styled modal.
- Events page shows raw Supabase error messages in the UI.
- Success screen after submitting shows a raw UUID as "Reference".
